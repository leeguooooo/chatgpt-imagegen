# Proactive Document Illustration — Design

**Date:** 2026-06-22
**Status:** Approved (design)
**Scope:** `SKILL.md` only (agent guidance). No code changes.

---

## Problem

`chatgpt-imagegen` works well, but agents rarely reach for it on their own. Its
`SKILL.md` triggering is **reactive** — "the user asks for an image." When an agent is
authoring a document, blog post, or technical proposal, it almost never proactively
illustrates the concepts, even though figures would help. The user wants illustration
to be **brought out automatically** during long-form writing, generated by **background
subagents** so the writing isn't blocked.

This is fundamentally a **SKILL.md change**, not a code feature: skills are pull-based —
the host agent (Claude Code, Codex, etc.) reads the skill's frontmatter `description`
and "When to use" and decides whether to invoke. We cannot force auto-triggering; we can
only make the triggering language proactive and give a clear illustration workflow. The
"prompt optimization" the user mentioned is likewise written guidance, not code — the
host is already an LLM and can rewrite prompts itself; it just needs to know how.

## Goals

- Make agents **proactively** propose and generate illustrations when authoring
  long-form explanatory content (docs, blogs, technical proposals, design docs).
- Keep it **transparent**: announce a brief figure plan, then generate — don't generate
  silently, don't block on approval.
- Generate via **background subagents** (one per figure) so writing continues.
- Respect the **user's backend configuration** for how parallel that actually is.
- Give concrete **figure-prompt guidance** so the generated images are useful.
- Guard against **over-illustration** and quota waste.

## Non-goals (YAGNI)

- No code changes — no batch CLI mode, no `--prompt-optimize` flag, no LLM prompt-rewriter.
- The skill does **not** pick or force a backend (web vs codex) — that is the user's
  configuration decision (`--backend` / `CHATGPT_IMAGEGEN_BACKEND`, default `auto`).
- The skill does **not** bind a default illustration style — style is chosen per document.
- No change to the CLI's behavior, flags, or tests.

---

## Key constraints (must be reflected in the guidance)

1. **Skills are pull-based.** The frontmatter `description` is the primary trigger signal
   a host indexes on. Strengthening it is the highest-leverage change for "auto-trigger."
2. **Parallelism vs. quota is the user's call.** The `web` backend has a concurrency cap
   of **1** (shared Chrome + page rate-limits) — N background subagents on `web` **queue**
   (safe, wall-clock ≈ serial sum). The `codex` backend allows **4** concurrent but bills
   the metered **Codex-usage** bucket. The skill must **not** override the user's
   `--backend` choice; it states the trade-off so the agent/user understands why "parallel"
   may actually be serialized.
3. **Quota is shared and real.** Proactive generation spends subscription quota and writes
   files. The "announce a plan first" step is the user's stop point; the anti-over-
   illustration guardrail keeps volume sane.

## Changes to `SKILL.md`

### A. Frontmatter `description` — add a proactive trigger clause

Append to the **end of the frontmatter `description:` YAML string** (after its current
final clause "…or extending an established repo icon system."). The `description` is one
dense inline paragraph in the YAML front matter — not the body's "When to use" H2:

> *Also use **proactively**: when authoring a document, blog post, technical proposal,
> design doc, README, or other long-form explanatory content, propose illustrations for
> the key concepts and generate them as background tasks — don't wait to be asked for an
> image.*

This is the single most important edit — it is what biases a host agent to invoke the
skill mid-writing.

### B. "When to use" — add proactive bullets

Add bullets for: authoring long-form/explanatory content (blog, tech proposal, design
doc, tutorial, postmortem, README) where a figure would make a concept land; and an
explicit "you don't need to be asked — propose figures and generate them" note.

### C. New section: "Illustrating documents" (the workflow)

Placed near the existing "Workflow" section. Contents:

1. **Announce a brief plan first** — one or two lines naming where figures go and what
   each depicts — then generate (don't ask for approval; the plan is the user's stop point).
2. **Fan out background subagents** — one per figure, each invoking the CLI with
   `--quiet -o <path>`; keep writing the prose meanwhile; embed each image when ready.
   The skill describes the *pattern*; spawning background tasks is the host agent's job
   (its Agent/Task tooling).
3. **Parallelism depends on the user's backend** — state the web=serialized /
   codex=parallel-but-billed trade-off; do not override `--backend`; default `auto`.
4. **Choose a style to fit the document's tone** — do not default a style. For informal
   or blog-style explainers the built-in `doodle` look fits (deliberately crude,
   content-accurate); for formal specs pick a cleaner look. Use `--style NAME` or a
   user-defined style (see the Styles section).
5. **Anti-over-illustration guardrail** — at most one figure per major concept; don't
   illustrate decoratively; never loop generating "variants of the same figure" (that
   just burns quota); iterate the prompt instead if a figure is wrong.

### D. New section (or subsection): "Writing figure prompts" (the "prompt optimization")

Concrete guidance — written, not code:

- Describe the figure's **content** precisely: the boxes, arrows, labels, layout, and
  relationships — not just a noun ("an architecture diagram" → say what's in it).
- One subject, one concept per figure.
- Name the desired **style** explicitly.
- For the `doodle` look: **content accuracy beats polish** — it's meant to look crude,
  but the information must still be readable.

### E. Housekeeping

Bump `SKILL.md` frontmatter `version: "0.7.0"` → `"0.11.0"` to align with the CLI.

## Acceptance / done criteria

- `SKILL.md` frontmatter `description` contains the proactive-illustration clause.
- "When to use" has proactive bullets for long-form authoring.
- An "Illustrating documents" section documents: announce-plan-then-generate, background
  fan-out, the backend/parallelism trade-off (no backend forced), style-per-tone (no
  default style bound), and the anti-over-illustration guardrail.
- A "Writing figure prompts" section gives the content-precise, single-subject,
  named-style, doodle-content-over-polish guidance.
- Frontmatter `version` is `0.11.0`.
- No changes to `chatgpt-imagegen` or its tests.
